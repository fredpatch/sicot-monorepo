### Server — new `packages/server/src/modules/analytics/`

- `analytics.types.ts` — one interface per module: `AccordsAnalytics`,
  `CourriersAnalytics`, `MissionsAnalytics`, `TraductionAnalytics`,
  `DemandesAnalytics`, `DocumentsAnalytics`, `GlossaireAnalytics`, plus a
  cross-module `GlobalAnalytics` (headline totals per module)
- `services/analytics.service.ts` (651 lines) — one `get*Analytics(filtre)`
  per module, each wrapped in `avecCache()` (see below), each defaulting to
  a trailing-12-months window (`resoudrePeriode()`) when no explicit
  `dateDebut`/`dateFin` is passed. Highlights:
  - Accords: durée moyenne par type, taux de renouvellement, répartition
    géographique, évolution mensuelle
  - Courriers: volume par mois/direction, temps moyen de réponse, taux de
    réponse, top organisations expéditrices, **évolution de la criticité**
    (reads from the new `courriersCriticiteSnapshots` table — see below)
  - Missions: par pays, état des recommandations, délai moyen de dépôt de
    rapport, top participants
  - Traduction: volume mensuel, taux de correction IA, temps de
    traitement, répartition par direction, termes ajoutés au glossaire
    depuis M6
  - Demandes: délai moyen de prise en charge, taux d'urgence validée,
    volume par demandeur, taux de complétion
  - Documents: volume par mois/catégorie, taux de succès OCR, évolution du
    stock total
  - Glossaire: croissance mensuelle, origine (manuel vs auto M6),
    répartition par domaine
- `controllers/analytics.controller.ts` (96 lines) — one thin handler per
  module + `parsePeriode()` which pushes `dateFin` to `23:59:59.999 UTC` so
  a date-only query string doesn't silently exclude "today"
- `routes/analytics.route.ts` — `GET /accords`, `/courriers`, `/missions`,
  `/traductions`, `/demandes`, `/documents`, `/glossaire`, `/global`. Gated
  by `authenticate` + `requireTraducteur` (role hierarchy: traducteur and
  above — excludes plain `agent`, unlike the portal's admin-only toggle)
- Mounted in `index.ts` as `app.use('/api/analytics', analyticsRoutes)`
- `utils/error.ts` — `handleAnalyticsError` added (empty error map, so any
  service error falls through to the generic 500 handler — no
  analytics-specific error codes exist yet)

### New: in-memory analytics cache (`packages/server/src/utils/cache.ts`)

`avecCache(cle, ttlMs, calculer)` — a plain `Map` with a 60s TTL
(`CACHE_TTL_MS` in analytics.service.ts), keyed by
`analytics:{module}:{debut.toISOString()}:{fin.toISOString()}`. Explicitly
scoped as "good enough for one process, revisit with Redis if the
deployment ever goes multi-instance" — a comment, not a config option.

### New: daily criticité-courriers snapshot (supports the M4 evolution chart)

Courrier criticité (normal/à surveiller/critique) is computed on-the-fly
from `parametres` thresholds and was **never persisted** anywhere — so
there was no way to chart it over time. Fixed with:

- Schema: new `courriersCriticiteSnapshots` table (`date` unique, `normal`,
  `aSurveiller`, `critique`, `totalEnAttente`, `createdAt`)
- `jobs/criticite-snapshot.ts` — `snapshotCriticiteCourriers()` reproduces
  `calculerCriticite()`'s logic in raw SQL (one query with `FILTER`
  clauses) instead of iterating rows in JS; upserts via
  `onConflictDoUpdate` on `date` so a manual re-run same-day overwrites
  rather than duplicates. `demarrerJobSnapshotCriticite()` schedules it
  daily at 23:55 via `node-cron`
- Registered in `index.ts` (`demarrerJobSnapshotCriticite()` at startup)
  **and** in `jobs/registre.ts` as a manually-triggerable job
  (`courriers_criticite_snapshot`, module M11, `roleMinimum: admin`) — the
  registry entry's own description notes it's useful in dev to backfill
  history without waiting several real days
- Caveat inherent to the design: history only starts accumulating from
  first deploy of this job — no way to backfill genuinely past dates

## Sprint 11 — Rapports layer (M11, second half) + demo data seed

Closes out Sprint 11's remaining scope: on-demand + monthly-automatic
report generation (PDF/Excel), analytics CSV/Excel export, and a
non-production data seeder to make the analytics dashboards meaningful to
look at without waiting months for real data to accumulate.

### Server — new `packages/server/src/modules/report/`

- `services/rapports.service.ts` (237 lines) — `genererRapport(params)`:
  collects data for the requested modules via the same
  `SERVICE_PAR_MODULE` map used by the export endpoint (new export from
  `analytics.service.ts`), renders either a combined multi-section PDF
  (`genererPDFRapport`, one page-break per module, reusing Sprint 10's
  `genererPDFDepuisHTML`) or a multi-sheet Excel workbook
  (`genererExcelRapport`, one sheet per metric, sheet names deduplicated
  and truncated to Excel's 31-char limit), writes the file to
  `DOSSIERS.rapport`, and archives it as a `documents` row (new `rapport`
  category) + a `rapports` row (history/audit trail). Reports generated
  by the cron (no `userId`) are attributed to the first `super_admin`
  found (`resoudreUtilisateurSysteme()`) since `documents.uploadePar` is
  `NOT NULL`. Also `listerRapports()` for the history view.
- `controllers/rapports.controller.ts` (44 lines) — `genererRapport`
  (validates `periodeDebut`/`periodeFin`/`modules`/`format`),
  `listerRapports`
- `routes/rapports.route.ts` — **created empty (0 bytes) and never
  filled in or imported anywhere.** The actual HTTP routes
  (`POST /api/analytics/rapports`, `GET /api/analytics/rapports`) are
  registered directly in `analytics.route.ts`, which imports
  `rapportsController` from the report module directly. This file is dead
  weight — should be deleted or actually used, not left as a confusing
  empty stub
- `jobs/rapport-mensuel.ts` — `genererRapportMensuel()` computes the
  previous calendar month's date range and calls `genererRapport()` twice
  (PDF + Excel, all 8 modules, `type: 'mensuel'`); `demarrerJobRapportMensuel()`
  schedules it via cron at `0 6 1 * *` (1st of month, 06:00). Started in
  `index.ts`. Also registered as a manually-triggerable job in
  `registre.ts` (`rapport_mensuel`, module M11, admin) — closes the
  Sprint 10 backlog item "Job manuel rapport mensuel" (`docs/TASKS.md`
  updated to reflect this)
